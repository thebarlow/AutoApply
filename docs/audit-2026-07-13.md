## Security

### S1 (P1) â€” Global `config`-table prompt endpoints are cross-tenant read/write
`web/routers/config.py` `/api/config/prompts/*`, plus `latex_templates`,
`active_*_prompt_id`, and `named_providers`, all go through `_get_global` /
`_set_global`, which hit the **global** `Config` table with no `profile_id`
scoping. Any authenticated tenant can list, edit, or delete these values and the
change lands for every tenant. The per-tenant split (Task 4) covered
scoring/templates/sources/search/job_searches but left the prompt endpoints on
the global key.

- **Impact:** depends on whether hosted generation reads these keys. Live
  per-tenant generation uses the `prompts` table via `user.resolve_prompt()`, so
  today this looks like a dormant/legacy surface rather than an active RCE-ish
  path â€” but it is still a cross-tenant write primitive and `latex_templates`
  stores **filesystem paths** that later get `Path(...).exists()`-checked and
  read. Confirm no code path renders from `latex_templates` before downgrading.
- **Fix:** either scope these to `profile_config` (like Task 4 did for the
  others) or, if they are truly dead in hosted mode, delete the endpoints and the
  global keys. See R1.

### S2 (P1) â€” Unmetered LLM call in `draft_section_prompt`
`web/routers/config.py:1317` `POST /api/config/section-prompt/draft` calls
`_llm_json_with_retry(...)` directly with **no `meter_action` wrapper**. Every
other tenant-facing LLM entry point (score/generate/eval/refine/extract) is gated
and debited; this one lets any authenticated tenant spend the platform LLM key
for free, with no credit floor check. Low per-call cost, but it is an
unauthenticated-against-balance drain and an inconsistency in the metering
invariant.
- **Fix:** wrap the call in `meter_action(db, profile_id, action="draft", ...)`,
  or explicitly document it as a free dry endpoint and add a per-tenant rate
  limit.

### S3 (P2) â€” `is_onboarding` trusted from client in `parse_apply`
`web/routers/config.py:979`. The code already carries an inline GUARD-RAIL
comment: `proposal.is_onboarding` comes from the client and the onboarding branch
**replaces the stored `profile_tree` wholesale**. Safe today only because intake
fires on empty profiles. Any future re-parse UI on a populated profile makes a
stray `is_onboarding=true` a full-profile-wipe primitive.
- **Fix (do before any re-parse UI ships):** re-derive `is_onboarding`
  server-side from stored data (as `parse_propose` already does) instead of
  trusting the payload.

### S4 (P3) â€” `require_admin` resolves admin via the *impersonated* tenant
`web/routers/credits.py:23` `require_admin` depends on `current_profile_id`,
which during impersonation returns the impersonated profile. So `admin_grant`,
`admin_set_tier`, `system_balance`, and `dev.py` resume-compare authorize against
the impersonated account, not the real admin. This **fails closed** (impersonating
a non-admin â†’ 403) and unsafe methods are already blocked by the impersonation
read-only middleware, so it is not an escalation â€” but it is inconsistent with
`admin.py`'s `require_real_admin`, which was written precisely to resolve the real
logged-in admin. Standardize on `require_real_admin` to avoid a future footgun.

### S5 â€” Reviewed, no action
- **Auth gate** (`web/auth/middleware.py`): `_EXEMPT_PATHS` is minimal and
  correct â€” webhook (own signature check), `stage-job` (own bearer token via
  `bearer_or_session_profile`), `ext/me`. Exact-match, not prefix, so no
  `/api/payments/webhook/../` bypass.
- **Stripe webhook** (`payments.py:180`): signature verified via
  `construct_event`; idempotency is an atomic conditional `UPDATE ... WHERE
  status != 'completed'` â€” a single payment cannot be double-credited even under
  a webhook/verify race. `verify` checks `purchase.profile_id == caller`. Solid.
- **Credit amounts** are server-computed from `resolve_price_id(tier)`; the client
  only sends `price_id`. Grant budget is capped against the OpenRouter system
  balance. Good.
- **SSRF in scrapers** (`scraper/search.py`): sources are the fixed Remotive /
  RemoteOK endpoints; user input is only a keyword/exclude/location filter, never
  a URL. No SSRF surface.
- **Path traversal** (`docs_router.py`, `config.py` file serving): docs endpoint
  rejects `/`, `\`, and non-`.md`; profile/job file serving reads DB-stored paths
  scoped by the caller's `profile_id`, never a client-supplied path.
- **Tenant scoping**: reads go through `Job.get(..., profile_id)` /
  `scoped()`; profile endpoints all guard `profile_id != caller_id â†’ 404`.
- **Secret handling**: `_write_env` is disabled in production (raises 400); keys
  come from platform env. `_validate_api_key` rejects newline injection.

### S6 (P1) â€” SSE broadcast + output files leak across tenants (found during I2)
Two cross-tenant leaks the request/response sweep missed, both rooted in treating
`job_key` (unique only per profile) as global:
- **SSE stream â€” FIXED 2026-07-13.** `/api/events` was a global broadcast; every
  `job` event pushed a full tenant-private payload to all connected clients, and
  the recipient's `upsertJob` (keyed on `job_key`) could overwrite its own job row
  with another tenant's. Fix: `sse.subscribe`/`send` carry a `profile_id`;
  `/api/events` subscribes with the caller's tenant; `job`/`credits`/`prompt_reset`
  and the `llm_status`/`llm_action` events (plus the `llm_status` registry and its
  `/api/llm-status` seed) are all tenant-scoped now.
- **Output files â€” FIXED 2026-07-13.** `generator/outputs/{job_key}_...` had no
  tenant dimension: two tenants sharing a `job_key` wrote the same PDF/MD; the
  loser's `serve_resume` (passes the `Job.get` scope check but reads the shared
  file) returned the wrong tenant's document. Every built path is now prefixed
  with the profile (`{profile_id}_{job_key}_{name}`). Stored `resume_path`/
  `cover_path` are absolute so already-rendered docs keep serving; old files
  orphan harmlessly (`documents` is the source of truth) and re-render into the
  new path â€” no prod migration required.
