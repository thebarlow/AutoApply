## Goal

Build **Model 2** ‚Äî a per-section, schema-driven r√©sum√© generator ‚Äî as a **lightweight, dev-only comparison harness** against **Model 1**, the current single-call `ResumeGeneration` path. For a given `job_key`, generate the r√©sum√© both ways, render each to Markdown, run the existing eval scorer on each, and show them **side by side** in a dev page. The user eyeballs results, decides which is better, and the winner is productionized in a later sub-project.

Model 2 is also the vehicle for the **original phase-3 capability**: it is the first path that authors content for **custom (user-defined) sections** added in 2C. Model 1 cannot; on custom sections Model 2 is the sole producer, which itself demonstrates the capability.

**Scope: r√©sum√© only.** Cover letters stay on their current single-call path and are out of scope.

## Field-Role Taxonomy (the schema-driven core)

Every `FieldNode` already carries `llm_input`, `llm_output`, `llm_instructions`, `regen_lock` (in `core/profile_tree.py`); today these are inert (they round-trip but drive nothing). Phase 3 activates them via a three-role model:

| Role | Behavior | Field attrs |
|---|---|---|
| **LLM-outputable** | Regenerated by the LLM from `llm_instructions` + section context. Rendered. `regen_lock=True` ‚Üí keep the current value this run (still rendered). | `llm_output=True` (`llm_instructions` is the per-field prompt) |
| **Immutable** | Rendered **verbatim**; never regenerated. Also handed to the LLM as context so outputable siblings stay accurate (e.g. company/title/dates anchor the experience bullets). | `llm_output=False`, `llm_input=False` |
| **Context-only** | Fed to the LLM as background; **not rendered** on the r√©sum√© (e.g. private notes about a role). | `llm_input=True`, `llm_output=False` |

`regen_lock` is a per-field user "pin": it overrides outputable for one generation run, leaving the current value untouched.

A section's generation context = its **immutable** + **context-only** fields (plus the job context). Its outputs = its unlocked **outputable** fields.

## Architecture

Six isolated units. The editor change is pure frontend; generation/rendering are new `core/` modules; one new dev-only endpoint and one dev-only page.

### 1. Editor role control (frontend, no backend change)

The tree already round-trips `llm_input`/`llm_output`/`llm_instructions`/`regen_lock`, and the whole-tree `PUT /api/config/profiles/{id}/tree` already persists and validates them. So this is purely surfacing controls in the existing tree editor:

- In `react-dashboard/src/components/widgets/profile-tree/`, each field (`FieldView` in `TreeNode.jsx`, with new presentational pieces in `structuralControls.jsx`/`fieldWidgets.jsx` as needed) gains:
  - a **role selector** (Context-only / Immutable / LLM-outputable) that writes `llm_input` + `llm_output` per the taxonomy table;
  - a **regen-lock** toggle (`regen_lock`);
  - an **`llm_instructions`** text box, shown only when the role is outputable.
- Writes go through the existing `ops`/`updateNode` immutable-tree path and the existing dirty/save/`PUT` flow. New `treeOps` helper(s) (e.g. `setFieldRole`, `setLlmInstructions`, `toggleRegenLock`) are pure and unit-tested.
- `makeField` / built nodes must carry these attrs with defaults so they round-trip (already required by the 2C invariants).

### 2. `User.profile_tree_root()` (core/user.py)

Small accessor returning the stored tree (`user.data['profile_tree']`) as a validated `RootNode` (migrating a legacy profile via `legacy_to_tree` if no tree is stored, mirroring existing load behavior). Model 2 walks the real tree, not the legacy-flattened attrs.

### 3. `core/section_generator.py` (new, the meat)

Pure-ish module (no DB; takes an LLM client + model). Public surface:

- `generate_resume_by_section(root, job_ctx, client, model) -> dict[str, str | list[str]]`
  - Walks **visible** sections in tree order.
  - For each section that has at least one **unlocked outputable** field (directly, or inside a list's entries), makes **one focused LLM call**:
    - **Input:** the job context (`job_ctx`, the same extracted-description Markdown Model 1 uses), the section's immutable + context-only field values as anchored context, and each outputable field's `llm_instructions`.
    - **Output contract:** strict JSON keyed by node id. For list sections, every entry's outputable field is requested, keyed by `entry_id` ‚Üí `{field_key: value}`. Validated against a per-call Pydantic model and run through the existing `_llm_json_with_retry` hardening.
  - Returns a flat map: `field_node_id -> authored value` (string for `text`/`markdown`, list for `bullets`/`taglist`). Locked or non-outputable fields are absent (the renderer falls back to the stored value).
  - Sections with no outputable fields make **no** LLM call.

This is the schema-driven engine: it reads roles off the tree, not a hardcoded section list, so preset and custom sections are handled uniformly.

### 4. `core/tree_render.py` (new, throwaway ‚Äî clearly marked)

Pure module: `render_tree_markdown(root, authored) -> str`. Walks the tree, overlays the `authored` map onto outputable fields (stored value when absent/locked), and emits readable Markdown including custom sections. **Omits context-only fields.** Hidden (`visible=False`) sections/fields are skipped.

This is explicitly **not** the phase-4 production renderer ‚Äî its docstring and `core/CONTEXT.md` say so. It exists only to make Model 2 output readable for the comparison; the real schema-driven renderer is sub-project #4.

### 5. Dev endpoint `POST /api/dev/resume-compare/{job_key}`

New dev/admin-gated route (in `web/routers/`, e.g. `dev.py`; gated so it's unavailable to non-admin users in production). For the given job:

1. Resolves the user, job, LLM client + model (same resolution generation uses).
2. **Model 1 (dry):** `build_resume_prompt` ‚Üí `_llm_json_with_retry(ResumeGeneration)` ‚Üí `build_resume_document` ‚Üí `assemble_resume_markdown`. **No `Document.upsert`, no file write** ‚Äî nothing overwrites the job's real stored r√©sum√©.
3. **Model 2:** `generate_resume_by_section(...)` ‚Üí `render_tree_markdown(...)`.
4. Runs `evaluate_resume_md` on each rendered Markdown (objective score + issues).
5. Returns `{ model1: { markdown, score, issues }, model2: { markdown, score, issues } }`.

The harness costs LLM tokens but is **not** metered/debited and does **not** run the ATS gate (out of scope, per "lightweight").

### 6. Dev side-by-side page (frontend)

A dev-only page/route in the dashboard: input a `job_key`, call the endpoint, render the two Markdown r√©sum√©s in two columns with each side's eval score + issues. Minimal styling, reusing the existing Markdown rendering the dashboard already uses where practical.

## Error Handling

- A Model 2 section call that fails JSON validation retries once via `_llm_json_with_retry`; if it still fails, that section's outputable fields fall back to stored values and the failure is surfaced in the endpoint response (so the comparison still renders rather than 500-ing).
- Empty-content (token-limit) responses raise a clear message, mirroring Model 1's existing empty-content guard.
- The endpoint catches per-model failures independently: if Model 1 errors, Model 2 still returns, and vice versa, each with an `error` field.

## Testing

- **`section_generator`** (unit, stubbed LLM): per-section call construction; only unlocked outputable fields requested; `regen_lock` respected; immutable + context-only fed as context but not output; list keying by `(entry_id, field_key)`; sections with no outputable fields make no call; failed section falls back to stored values.
- **`tree_render`** (unit): preset + custom sections render; authored overlay applied; context-only omitted; hidden nodes skipped; locked outputable shows stored value.
- **`treeOps` role helpers** (Vitest): `setFieldRole`/`setLlmInstructions`/`toggleRegenLock` produce correct flag combinations immutably.
- **Editor role control** (Vitest/RTL): selecting a role updates the field node; `llm_instructions` box appears only for outputable.
- **Dev endpoint** (thin): with LLM + eval stubbed, returns both models' markdown/score; one model failing still returns the other.

Python: TDD, type hints, `black`, Google-style docstrings. Frontend: existing dashboard JS conventions + Vitest, `npm run test` from `react-dashboard/`.
